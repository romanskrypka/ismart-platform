## Задание 1: Разработка политики RBAC для AI-агентов (Базовый уровень)

**Цель:** Научиться определять роли и разрешения для AI-агентов на основе их бизнес-функций, применяя принцип наименьших привилегий.

**Сценарий:** Вы работаете в компании, которая разрабатывает AI-ассистента для отдела продаж. Ассистент должен помогать менеджерам по продажам в их ежедневной работе.

**Задачи ассистента:**
1.  Анализировать историю общения с клиентами (email, звонки) для подготовки к переговорам.
2.  Получать доступ к каталогу продуктов и актуальным ценам.
3.  Создавать и обновлять карточки клиентов в CRM-системе.
4.  Генерировать еженедельные отчеты по личной эффективности менеджера (количество звонков, встреч, закрытых сделок).

**Инструкция:**

1.  **Определите роли.** Подумайте, можно ли разделить функционал ассистента на несколько логических ролей, чтобы минимизировать риски. Например, «Аналитик коммуникаций», «Менеджер CRM», «Продуктовый консультант», «Генератор отчетов». Создайте как минимум 3 роли.
2.  **Определите ресурсы.** Перечислите все информационные ресурсы, к которым должны обращаться агенты для выполнения своих задач (например, база данных CRM, почтовый сервер, файловое хранилище с каталогом продуктов, API для отчетов).
3.  **Создайте матрицу доступа.** Нарисуйте в Markdown таблицу, где по строкам будут ресурсы, а по столбцам — роли. На пересечении укажите необходимые права доступа: `Read-Only` (только чтение), `Write` (запись/изменение), `Create` (создание), `Delete` (удаление), `No Access` (нет доступа).

**Ожидаемый результат:**

Вы должны предоставить Markdown-таблицу (матрицу RBAC), которая четко и однозначно определяет права доступа для каждой роли. Политика должна строго следовать принципу наименьших привилегий.

**Пример таблицы:**

| Ресурс / Роль | Аналитик коммуникаций | Менеджер CRM | Продуктовый консультант |
| :--- | :--- | :--- | :--- |
| База данных CRM (Карточки клиентов) | `Read-Only` | `Create`, `Write` | `Read-Only` |
| Почтовый сервер (История переписки) | `Read-Only` | `No Access` | `No Access` |
| ... | ... | ... | ... |

**Критерии оценки:**

*   **Полнота:** Определены все необходимые роли и ресурсы.
*   **Принцип наименьших привилегий:** Предоставлены только минимально необходимые права для каждой роли. Нет избыточных разрешений.
*   **Четкость:** Матрица легко читается и не допускает двойных толкований.

**Тайминг:** 25 минут.

## Задание 2: Анонимизация данных для внешней AI-модели (Продвинутый уровень)

**Цель:** Практически применить методы анонимизации для подготовки набора данных к безопасной передаче внешнему AI-сервису.

**Сценарий:** Ваша компания хочет использовать мощную внешнюю языковую модель (LLM) для анализа отзывов клиентов о вашем продукте. Отзывы содержат полезную информацию, но также и персональные данные клиентов (имена, email, иногда номера заказов).

**Пример фрагмента отзыва (сырые данные):**

```json
{
  "review_id": "rev-98765",
  "author_name": "Анна Петрова",
  "author_email": "anna.petrova@email.com",
  "order_id": "ord-12345",
  "rating": 5,
  "review_text": "Отличный сервис! Менеджер Иван Сидоров быстро помог решить мою проблему с заказом ord-12345. Обязательно буду рекомендовать вас друзьям. Мой email для связи: anna.petrova@email.com."
}
```

**Инструкция:**

1.  **Выберите методы анонимизации.** Определите, какие методы (псевдонимизация, маскирование, обобщение, замена сущностей) вы будете использовать для каждого типа персональных данных в отзыве.
2.  **Напишите псевдокод или опишите алгоритм.** Опишите по шагам, как будет работать ваш скрипт анонимизации. Какие данные и на что вы будете заменять? Например: "1. Найти все email-адреса с помощью регулярного выражения. 2. Заменить каждый найденный email на строку '[EMAIL REMOVED]'. 3. Найти все имена собственные и заменить их на '[PERSON NAME]'...".
3.  **Примените ваш алгоритм.** Вручную или мысленно примените ваш алгоритм к приведенному выше примеру отзыва и покажите, как будет выглядеть итоговый, анонимизированный JSON-объект, готовый к отправке во внешнюю модель.

**Ожидаемый результат:**

1.  Список выбранных методов анонимизации с обоснованием.
2.  Пошаговый алгоритм (или псевдокод) процесса анонимизации.
3.  Финальный JSON-объект с анонимизированными данными.

**Критерии оценки:**

*   **Безопасность:** Все персональные и чувствительные данные удалены или надежно обезличены.
*   **Сохранение ценности:** Текст отзыва после анонимизации сохранил свой основной смысл и остается полезным для анализа.
*   **Логичность:** Алгоритм анонимизации логичен, последователен и эффективен.

**Тайминг:** 35 минут.

## Задание 3: Проектирование политики аудита для AI-агента (Экспертный уровень)

**Цель:** Разработать детальную политику логирования для критически важного AI-агента, чтобы обеспечить прозрачность, подотчетность и возможность расследования инцидентов.

**Сценарий:** Вы отвечаете за безопасность AI-агента, который автоматически управляет доступами сотрудников к внутренним системам компании. Агент может предоставлять и отзывать права доступа на основе запросов, одобренных руководителями.

**Действия агента:**
*   Получает запрос на изменение прав доступа (например, "Предоставить доступ к системе X для сотрудника Y").
*   Проверяет, одобрен ли запрос руководителем сотрудника Y.
*   Подключается к системе управления доступами (например, Active Directory).
*   Выполняет команду на предоставление или отзыв прав.
*   Уведомляет сотрудника и его руководителя о результате.

**Инструкция:**

1.  **Определите ключевые события для аудита.** Составьте список всех значимых событий в жизненном цикле запроса, которые должны быть записаны в журнал аудита. Подумайте о каждом шаге, который выполняет агент.
2.  **Спроектируйте структуру лога.** Для каждого события определите, какая информация должна быть записана. Используйте структуру, отвечающую на вопросы: Кто? Что? Когда? Где? Как? (ID агента, Роль, Действие, Ресурс, Результат, Временная метка, Исходный IP, и т.д.).
3.  **Опишите пример записи в логе.** Для одного из ключевых событий (например, успешное предоставление доступа) приведите пример того, как будет выглядеть полная запись в журнале аудита в формате JSON.
4.  **Предложите правила для алертов.** Опишите 2-3 правила для системы проактивного мониторинга. Какие аномальные события должны вызывать немедленное оповещение службы безопасности? (Например, "Попытка предоставить права без одобрения руководителя" или "Более 5 неудачных попыток подключения к системе за 1 минуту").

**Ожидаемый результат:**

1.  Список событий для аудита.
2.  Описание полей (структуры) для записи в логе.
3.  Пример записи в логе в формате JSON.
4.  Список правил для системы оповещения.

**Критерии оценки:**

*   **Детализация:** Политика аудита охватывает все критические этапы работы агента.
*   **Информативность:** Структура лога содержит всю необходимую информацию для расследования инцидентов.
*   **Проактивность:** Предложенные правила для алертов являются релевантными и помогают предотвращать инциденты, а не только реагировать на них.
*   **Структурированность:** Ответ четко структурирован и легко читаем.

**Тайминг:** 40 минут.
