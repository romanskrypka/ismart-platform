# Работа с локальными LLM: Полное руководство

Настоящее руководство представляет собой подробный материал по развертыванию, настройке и использованию больших языковых моделей (LLM) на локальном оборудовании. Локальный инференс обеспечивает полную конфиденциальность данных, отсутствие абонентской платы и возможность работы без подключения к интернету, что критически важно для корпоративных сред и разработки AI-агентов [1].

## 1. Инструменты и фреймворки

В 2026 году экосистема инструментов для локального запуска LLM достигла высокой степени зрелости. Выбор конкретного решения зависит от задач, доступного оборудования и требований к пользовательскому интерфейсу.

| Инструмент | Описание | Основные преимущества | Недостатки |
| :--- | :--- | :--- | :--- |
| **Ollama** | Консольный инструмент и API-сервер для управления и запуска LLM. | Простая установка одной командой, богатая библиотека моделей, встроенный API, совместимый с OpenAI. | Отсутствует встроенный графический интерфейс. |
| **LM Studio** | Кроссплатформенное десктопное приложение с графическим интерфейсом. | Удобный поиск и загрузка моделей из Hugging Face, интуитивно понятный чат, встроенный локальный сервер. | Проприетарное ПО, потребляет больше ресурсов. |
| **llama.cpp** | Низкоуровневая библиотека на C/C++ для инференса моделей LLaMA. | Максимальная производительность, минимальные зависимости, поддержка квантования (GGUF). | Высокий порог входа, требует компиляции или работы через CLI. |
| **Jan AI** | Open-source десктопное приложение с фокусом на приватность. | Полностью открытый исходный код, хранение данных локально, простой интерфейс. | Меньше функций по сравнению с LM Studio. |
| **AnythingLLM** | Полноценное приложение для работы с документами (RAG). | Встроенная поддержка загрузки документов, многопользовательский режим. | Требует настройки базы данных и векторного хранилища. |

Для большинства задач разработки и интеграции с AI-агентами **Ollama** является индустриальным стандартом благодаря своей простоте, стабильности и наличию мощного API [2].

## 2. Установка и настройка LLM с локальным инференсом

В данном разделе рассматривается установка платформы **Ollama**, так как она предоставляет наилучший баланс между производительностью и удобством использования для разработчиков.

### Установка Ollama

Процесс установки зависит от используемой операционной системы. Инструкции предполагают кроссплатформенный подход к разработке [3].

**macOS**
Загрузите официальный инсталлятор с сайта `ollama.com` или используйте пакетный менеджер Homebrew:
```bash
brew install ollama
```

**Windows**
Загрузите исполняемый файл `OllamaSetup.exe` с официального сайта и следуйте инструкциям мастера установки. Ollama автоматически добавится в системный PATH.

**Linux**
Для установки на Linux-системах используется официальный скрипт:
```bash
curl -fsSL https://ollama.com/install.sh | sh
```
Скрипт автоматически скачает необходимые бинарные файлы и настроит Ollama как системную службу `systemd`.

### 2.1. Полезные команды и интерфейсы

После установки Ollama управление моделями осуществляется через интерфейс командной строки (CLI).

| Команда | Описание | Пример использования |
| :--- | :--- | :--- |
| `ollama run <model>` | Скачивает (если необходимо) и запускает модель в интерактивном режиме чата. | `ollama run llama3.2` |
| `ollama pull <model>` | Загружает модель из реестра без её запуска. | `ollama pull qwen2.5` |
| `ollama list` (или `ls`) | Выводит список всех загруженных локально моделей и их размер. | `ollama ls` |
| `ollama ps` | Показывает модели, которые в данный момент загружены в оперативную память (RAM/VRAM). | `ollama ps` |
| `ollama rm <model>` | Удаляет модель с локального диска для освобождения места. | `ollama rm mistral` |
| `ollama serve` | Запускает Ollama в режиме фонового сервера (API). | `ollama serve` |

Для многострочного ввода в интерактивном чате используйте тройные кавычки `"""` [2].

### 2.2. Дополнительные настройки

Настройка Ollama осуществляется через переменные окружения. Это позволяет изменить сетевые параметры, расположение моделей и параметры инференса [4].

**Ключевые переменные окружения:**
*   `OLLAMA_HOST`: Изменяет адрес и порт, на котором работает API. По умолчанию это `127.0.0.1:11434`. Для доступа из локальной сети установите значение `0.0.0.0:11434`.
*   `OLLAMA_MODELS`: Переопределяет директорию для хранения загруженных моделей (полезно, если на системном диске мало места).
*   `OLLAMA_KEEP_ALIVE`: Устанавливает время удержания модели в памяти после последнего запроса (по умолчанию 5 минут). Значение `-1` оставляет модель в памяти навсегда.
*   `OLLAMA_CONTEXT_LENGTH`: Задает размер контекстного окна в токенах (по умолчанию 4096).

**Применение настроек:**
*   **Windows:** Настройки -> Система -> О системе -> Дополнительные параметры системы -> Переменные среды.
*   **macOS:** Использование команды `launchctl setenv OLLAMA_HOST "0.0.0.0:11434"` с последующим перезапуском приложения.
*   **Linux:** Редактирование файла службы через `sudo systemctl edit ollama.service` и добавление блока `[Service]` с директивой `Environment="OLLAMA_HOST=0.0.0.0:11434"`.

### 2.3. Удаление локальной платформы LLM

Корректное удаление Ollama включает деинсталляцию самого приложения и удаление загруженных моделей, которые могут занимать десятки гигабайт.

**Windows**
Используйте стандартный инструмент "Установка и удаление программ" в настройках Windows. Модели хранятся в `C:\Users\%username%\.ollama\models` и должны быть удалены вручную, если они не были удалены деинсталлятором.

**macOS**
Удалите приложение из папки `/Applications`. Затем выполните в терминале:
```bash
sudo rm /usr/local/bin/ollama
rm -rf ~/.ollama
```

**Linux**
Остановите и удалите службу, затем удалите бинарные файлы и пользователя:
```bash
sudo systemctl stop ollama
sudo systemctl disable ollama
sudo rm /etc/systemd/system/ollama.service
sudo rm $(which ollama)
sudo rm -rf /usr/share/ollama
sudo userdel ollama
sudo groupdel ollama
```

## 3. Установка клиента с графическим интерфейсом пользователя (GUI)

Для комфортной работы с локальными моделями рекомендуется использовать графические интерфейсы. Одним из самых мощных и популярных решений является **Open WebUI** [5]. Это веб-интерфейс, который визуально и функционально напоминает ChatGPT, но работает локально и полностью интегрируется с Ollama.

В соответствии с требованиями к разработке (например, при ограничениях на использование Docker), Open WebUI можно установить напрямую через менеджер пакетов Python `pip` [6].

**Шаги по установке Open WebUI без Docker:**

1.  Убедитесь, что у вас установлен Python 3.11 (рекомендуемая версия).
2.  Создайте и активируйте виртуальную среду (подробнее в разделе 4).
3.  Установите пакет Open WebUI:
    ```bash
    pip install open-webui
    ```
4.  Запустите сервер Open WebUI:
    ```bash
    open-webui serve
    ```
5.  Откройте браузер и перейдите по адресу `http://localhost:8080`.

Open WebUI автоматически обнаружит запущенный экземпляр Ollama (на порту 11434) и подтянет список доступных моделей. Первый зарегистрированный пользователь автоматически становится администратором системы.

### 4.4. SQLite

При разработке AI-агентов часто возникает необходимость локального хранения данных (история чатов, логи взаимодействий, метаданные документов). **SQLite** — это легковесная реляционная база данных, которая хранит все данные в одном файле на диске.

SQLite не требует отдельного процесса сервера или сложной настройки. Модуль `sqlite3` встроен в стандартную библиотеку Python.

**Пример базового использования в Python:**
```python
import sqlite3

# Подключение к БД (файл будет создан автоматически, если не существует)
conn = sqlite3.connect('agent_memory.db')
cursor = conn.cursor()

# Создание таблицы
cursor.execute('''
    CREATE TABLE IF NOT EXISTS chat_history (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        role TEXT NOT NULL,
        content TEXT NOT NULL,
        timestamp DATETIME DEFAULT CURRENT_TIMESTAMP
    )
''')

# Добавление записи
cursor.execute("INSERT INTO chat_history (role, content) VALUES ('user', 'Привет, локальная LLM!')")
conn.commit()
conn.close()
```

## 5. Создание первого приложения с локальной LLM

Интеграция локальных моделей в собственные приложения — ключевой навык для AI-разработчика. Официальная библиотека `ollama` для Python предоставляет удобный интерфейс, аналогичный API от OpenAI.

В этом примере мы создадим простого консольного ассистента с памятью контекста (многошаговый диалог) и потоковым выводом ответа (streaming), что создает эффект "печатания" текста в реальном времени.

**Шаг 1. Установка библиотеки**
Убедитесь, что вы находитесь в активированной виртуальной среде (например, `ai_env`), и установите пакет:
```bash
pip install ollama
```

**Шаг 2. Написание кода приложения**
Создайте файл `app.py` и добавьте следующий код:

```python
import ollama

def main():
    print("🤖 Локальный AI-ассистент запущен. Введите 'выход' для завершения.")
    
    # Инициализация истории сообщений с системным промптом
    messages = [
        {
            'role': 'system',
            'content': 'Ты полезный, умный и лаконичный AI-ассистент. Отвечай на русском языке.'
        }
    ]
    
    # Используемая модель (убедитесь, что она загружена: ollama pull llama3.2)
    model_name = 'llama3.2'
    
    while True:
        user_input = input("\nВы: ")
        if user_input.lower() in ['выход', 'exit', 'quit']:
            break
            
        # Добавляем сообщение пользователя в историю
        messages.append({'role': 'user', 'content': user_input})
        
        print("AI: ", end="", flush=True)
        
        try:
            # Запрос к Ollama с включенным streaming
            stream = ollama.chat(
                model=model_name,
                messages=messages,
                stream=True
            )
            
            assistant_reply = ""
            for chunk in stream:
                content = chunk['message']['content']
                print(content, end="", flush=True)
                assistant_reply += content
                
            # Добавляем полный ответ ассистента в историю для сохранения контекста
            messages.append({'role': 'assistant', 'content': assistant_reply})
            print() # Перенос строки после завершения ответа
            
        except Exception as e:
            print(f"\n[Ошибка]: {e}")
            print("Убедитесь, что сервер Ollama запущен (ollama serve).")

if __name__ == "__main__":
    main()
```

**Шаг 3. Запуск приложения**
```bash
python app.py
```
Это приложение демонстрирует базовый паттерн взаимодействия с LLM: формирование списка сообщений (`messages`), отправку запроса и потоковую обработку ответа.

## 6. Устранение ошибок

При работе с локальными LLM могут возникать специфические ошибки. Ниже приведены наиболее частые проблемы и способы их решения [10].

### Ошибка: `bind: address already in use` или `Error: listen tcp 127.0.0.1:11434: bind`
**Причина:** Порт 11434 уже занят другим процессом (чаще всего — другой запущенной копией Ollama).
**Решение:** 
*   **Linux/macOS:** Найдите процесс командой `lsof -i :11434` или `sudo ss -ltnp | grep 11434` и завершите его с помощью `kill -9 <PID>`.
*   **Windows:** В PowerShell выполните `netstat -ano | findstr :11434`, найдите PID и завершите процесс через Диспетчер задач.
*   Если вы используете Ollama как службу, просто используйте `sudo systemctl restart ollama`.

### Ошибка: `Error: pull model manifest: file does not exist`
**Причина:** Попытка запустить модель, которая не скачана на локальный диск, или опечатка в названии модели.
**Решение:** Проверьте правильность написания названия модели на сайте ollama.com/library и выполните предварительное скачивание: `ollama pull <model_name>`.

### Ошибка: `CUDA error: out of memory` (OOM)
**Причина:** Видеокарте не хватает VRAM (видеопамяти) для загрузки модели или обработки контекста.
**Решение:**
1.  Закройте другие приложения, использующие GPU (браузеры с аппаратным ускорением, игры, графические редакторы).
2.  Уменьшите размер контекстного окна через переменную `OLLAMA_CONTEXT_LENGTH` (например, установите 2048 вместо 4096).
3.  Используйте модель меньшего размера (например, 3B или 8B вместо 14B) или с более сильным квантованием (Q4 вместо Q8).

### Проблема: Ollama работает очень медленно (на CPU вместо GPU)
**Причина:** Ollama не смогла обнаружить видеокарту или отсутствуют необходимые драйверы.
**Решение:**
*   Проверьте логи (на Linux: `journalctl -u ollama`). Если вы видите `msg="failure during GPU discovery"`, проблема в драйверах.
*   Убедитесь, что установлены последние драйверы NVIDIA или AMD ROCm (версии 7+ для Linux).
*   На Linux проверьте, что пользователь `ollama` добавлен в группы `video` и `render`: `sudo usermod -aG video,render ollama`.

## 7. Аппаратное ускорение

Производительность LLM измеряется в токенах в секунду (tokens/s). Для комфортной работы требуется скорость генерации от 10 до 20 токенов в секунду. Выбор аппаратного обеспечения критически влияет на этот показатель [11].

### 7.1. Рабочая станция с CPU

Инференс на центральном процессоре (CPU) — самый доступный, но и самый медленный способ. Скорость ограничена пропускной способностью оперативной памяти (RAM).

*   **Требования:** Для моделей 7B-8B (например, Llama 3) требуется минимум 8 ГБ RAM (рекомендуется 16 ГБ). Для моделей 13B-14B — от 16 до 32 ГБ RAM.
*   **Оптимизация:** Платформы вроде Ollama и llama.cpp автоматически используют системную память, если VRAM недоступна или её недостаточно.

### 7.2. Рабочая станция с GPU

Использование графического процессора (GPU) ускоряет генерацию в 10-50 раз по сравнению с CPU. Главный ограничитель — объем видеопамяти (VRAM), так как модель должна полностью поместиться в нее для максимальной скорости [11].

*   **NVIDIA:** Наилучшая поддержка (через CUDA). Поддерживаются видеокарты с Compute Capability 5.0+ (начиная с серии GTX 10xx, оптимально RTX 30xx/40xx). Для моделей 8B с квантованием Q4 достаточно 6-8 ГБ VRAM (RTX 3060/4060).
*   **AMD:** Поддерживается через библиотеку ROCm (на Linux) или Vulkan. Требуются современные карты (серии RX 6000/7000).
*   **Apple Silicon:** Процессоры M1/M2/M3/M4 с объединенной памятью (Unified Memory) показывают превосходные результаты через API Metal, так как GPU может использовать всю доступную системную память.

### 7.3. Включение AVX/AVX2 для ускорения работы процессора

Если вы вынуждены использовать CPU, критически важно наличие векторных инструкций **AVX** (Advanced Vector Extensions) или **AVX2**. Они позволяют процессору обрабатывать несколько блоков данных за одну инструкцию (SIMD), что ускоряет матричные вычисления нейросетей в 4-5 раз.

*   **Как проверить поддержку (Linux):** `grep avx2 /proc/cpuinfo`
*   **Как проверить поддержку (Windows):** Используйте утилиту CPU-Z.
*   Ollama и llama.cpp автоматически определяют наличие AVX/AVX2 и используют соответствующие оптимизированные бинарные файлы. Если ваш процессор старый и не поддерживает AVX (например, выпущен до 2013 года), инференс будет экстремально медленным, и некоторые современные сборки могут вообще не запуститься [12].

### 7.4. Использование сторонних платформ ASIC или VPS (российские) с поддержкой GPU

Для корпоративных задач или обучения моделей локального "железа" часто не хватает. Аренда облачных серверов (VPS) с GPU — стандартная практика. В условиях ограничений на зарубежные платежи актуальны российские провайдеры [13].

**Популярные российские провайдеры GPU-серверов:**
1.  **Selectel:** Предлагает облачные и выделенные серверы с картами от NVIDIA Tesla T4 (отлично для инференса) до мощных A100/H100. Высокая надежность (Tier III).
2.  **Hostkey:** Широкий выбор конфигураций, включая потребительские карты (RTX 3090, 4090), что значительно дешевле профессиональных Tesla/A-серий, но отлично подходит для LLM.
3.  **Cloud4Y / Timeweb Cloud:** Предоставляют VPS с GPU (vGPU) с почасовой тарификацией, что удобно для тестирования гипотез без долгосрочных обязательств.

**ASIC-платформы:** Специализированные чипы (например, Groq) обеспечивают невероятную скорость (сотни токенов в секунду), но они предоставляются только как облачное API и не подходят для полностью локального развертывания.

### 7.5. Использование сервиса Google Colab и Kaggle

Для обучения, экспериментов и прототипирования можно использовать бесплатные облачные Jupyter-ноутбуки, предоставляющие доступ к GPU.

**Google Colab**
*   Предоставляет бесплатный доступ к GPU NVIDIA T4 (16 ГБ VRAM).
*   Ограничения: сессии до 12 часов, внезапные отключения при неактивности, непостоянная доступность GPU в бесплатной версии.
*   **Запуск Ollama в Colab:** Возможен, но требует использования туннелирования (например, ngrok или pinggy) для проброса порта 11434 наружу, так как Colab не позволяет напрямую открывать порты. Дополнительно требуется запуск Ollama в фоновом процессе через `asyncio` [14].

**Kaggle Notebooks**
*   Более щедрые лимиты для ML-разработчиков: до 30 часов бесплатного использования GPU в неделю.
*   Предоставляет доступ к NVIDIA P100 или двум T4 (2x16 ГБ VRAM).
*   Отлично подходит для запуска более крупных моделей (до 14B-32B параметров при использовании двух T4).
*   Ограничения: сессии до 12 часов (до 9 часов в фоновом режиме), требует верификации по номеру телефона [15].

## 8. Разработка и использование ИИ-агентов

ИИ-агенты (AI Agents) представляют собой следующий эволюционный шаг в развитии искусственного интеллекта после обычных чат-ботов. Они способны не просто отвечать на вопросы, но и автономно действовать в цифровой среде для достижения поставленных целей [16].

### 8.1. Отличие ИИ-агентов от ИИ-инструментов

Разница между ИИ-инструментом (AI Tool) и ИИ-агентом заключается в уровне автономности и способе взаимодействия с задачей [17].

*   **ИИ-инструмент (AI Tool):** Требует явных пошаговых инструкций от человека. Он выполняет ровно то, что ему сказано в рамках одного запроса (single-pass). Если возникает ошибка, инструмент останавливается и ждет новых указаний. Пример: генератор изображений или стандартный ChatGPT, который просто отвечает на промпт.
*   **ИИ-агент (AI Agent):** Получает высокоуровневую цель (например, "Найди информацию о конкурентах и составь отчет") и самостоятельно планирует шаги для ее достижения. Агент работает в цикле (Agent Loop): он рассуждает, вызывает нужные инструменты, анализирует результаты их работы, исправляет собственные ошибки и продолжает действовать до полного выполнения задачи [18].

### 8.2. Классификация ИИ-агентов

В зависимости от сложности архитектуры и уровня автономности, ИИ-агенты делятся на несколько категорий:

1.  **Реактивные агенты (Simple Reflex Agents):** Действуют исключительно на основе текущих входных данных по заранее заданным правилам ("если-то"), не имея памяти о прошлых событиях.
2.  **Агенты с состоянием (Model-Based Reflex Agents):** Поддерживают внутреннюю модель мира и учитывают историю прошлых взаимодействий, что позволяет им справляться с частично наблюдаемыми средами.
3.  **Целеустремленные агенты (Goal-Based Agents):** Обладают информацией о желаемом результате и могут планировать последовательность действий (Plan-and-Execute) для его достижения, выбирая оптимальный путь.
4.  **Обучающиеся агенты (Learning Agents):** Способны улучшать свою производительность со временем, анализируя обратную связь от среды и корректируя свои алгоритмы принятия решений.
5.  **Многоагентные системы (Multi-Agent Systems):** Группы специализированных агентов (например, "исследователь", "писатель", "критик"), которые обмениваются информацией и сотрудничают для решения сложных комплексных задач.

### 8.3. Архитектура ИИ-агентов

Полноценный ИИ-агент состоит из множества взаимосвязанных компонентов, которые обеспечивают его автономность [19]. Для функционирования агента обязательными являются: системный промпт, оркестратор, доступ к инструментам и базовая модель (LLM или SLM).

#### 8.3.1. Промпт/команда (System Prompt)
Базовая инструкция, которая задает роль, персону (опционально), ограничения и глобальные правила поведения агента. Это "конституция", которой агент следует при принятии любых решений.

#### 8.3.2. Входной компонент (Perception)
Модуль восприятия, который принимает данные из внешней среды. Это может быть текстовый запрос пользователя, ответ от API, системная ошибка или результат выполнения предыдущего шага.

#### 8.3.3. Компонент обратной связи: Оценка, Самомониторинг
Механизм (часто называемый Reflection или Critic), позволяющий агенту оценивать промежуточные результаты. Если инструмент вернул ошибку или данные неполные, агент анализирует причину и корректирует свой следующий шаг, предотвращая зацикливание.

#### 8.3.4. Компонент планирования: Рабочий процесс, задачи
Модуль, отвечающий за декомпозицию сложной проблемы на выполнимые подзадачи. Использует техники вроде Chain of Thought (Цепочка мыслей) или ReAct (Reasoning and Acting) для построения логического маршрута.

#### 8.3.5. Компонент обучения: Модели тонкой настройки, Продолжение обучения
Механизмы адаптации агента. Включают как тонкую настройку (Fine-tuning) базовой модели под специфические задачи, так и методы непрерывного обучения (Continual Learning), позволяющие агенту усваивать новый опыт без полного переобучения (часто реализуется через пополнение векторной базы знаний).

#### 8.3.6. Компонент обработки (Мозг): LLM, система на основе правил
Ядро агента (Agent Core) — большая (LLM) или малая (SLM) языковая модель, которая выполняет роль "движка рассуждений". Именно она анализирует контекст, принимает решения и формирует команды для инструментов.

#### 8.3.7. Хранилище: Кратковременная память, Долговременная память, БД, Векторная БД
*   **Кратковременная память:** Хранит историю текущего сеанса или цепочку рассуждений для конкретной задачи (Train of Thought).
*   **Долговременная память:** Сохраняет знания между сессиями. Часто реализуется с помощью локальных реляционных БД (например, SQLite для метаданных) и векторных баз данных (ChromaDB, FAISS, Qdrant) для семантического поиска (RAG).

#### 8.3.8. Компонент действия: Инструменты: API, вебхуки, протоколы другие
Набор исполняемых функций (Tools), которые агент может вызывать. Это "руки" агента: веб-поиск (например, SearXNG), интерпретатор кода Python, доступ к файловой системе или внешним сервисам через Model Context Protocol (MCP).

#### 8.3.9. Компонент выхода (Output formatting)
Модуль, который форматирует результаты работы агента в удобный для пользователя или другой системы вид (JSON, Markdown, структурированный отчет).

#### 8.3.10. Выход (Action/Response)
Финальный результат работы: отправка сообщения пользователю, сохранение файла или передача данных другому агенту (например, через протокол A2A).

### 8.4. Фреймворки для разработки ИИ-агентов

Для создания агентов существует множество фреймворков, каждый из которых имеет свои сильные стороны [20]:

| Фреймворк | Описание и особенности | Лучшее применение |
| :--- | :--- | :--- |
| **LangGraph** | Мощная библиотека от создателей LangChain. Использует графовый подход для создания сложных циклических рабочих процессов с управлением состоянием (stateful). | Сложные enterprise-приложения, требующие точного контроля над каждым шагом агента. |
| **CrewAI** | Популярный Python-фреймворк для многоагентных систем. Агентам назначаются роли, цели и инструменты, после чего они работают как слаженная "команда". | Проекты, где задачу нужно разделить между несколькими специализированными ролями (например, исследователь и писатель). |
| **AutoGen** | Фреймворк от Microsoft, ориентированный на общение агентов друг с другом (conversational pattern). Поддерживает сложные паттерны взаимодействия. | Автоматизация написания кода и сложные многоагентные дебаты. |
| **smolagents** | Легковесная библиотека от Hugging Face. Фокусируется на `CodeAgent`, который выполняет действия, генерируя и запуская код Python. | Быстрое прототипирование, интеграция с локальными моделями (Ollama) и инструментами. |
| **Agno** | Открытый фреймворк (ранее Phidata) для создания "чистых" Python-агентов с памятью, инструментами и встроенной поддержкой векторных БД. | Разработка независимых локальных агентов с сохранением состояния. |

*Примечание: При разработке сложных многоагентных систем рекомендуется использовать связку Python, n8n для автоматизации рабочих процессов, Django для веб-интерфейса, а также SQLite для локальной разработки с последующим переходом на PostgreSQL в production.*

### 8.5. Пошаговое руководство по созданию ИИ-агента на практике

В этом примере мы создадим автономного агента-исследователя с использованием легковесного фреймворка `smolagents` и локальной модели через `Ollama`. Агент сможет самостоятельно искать информацию в интернете и анализировать ее [21].

**Шаг 1. Установка зависимостей**
Убедитесь, что Ollama установлена и модель загружена (например, `ollama pull qwen2.5-coder`).
```bash
pip install smolagents duckduckgo-search
```

**Шаг 2. Импорт библиотек и настройка модели**
Создайте файл `research_agent.py`:
```python
from smolagents import CodeAgent, DuckDuckGoSearchTool, LiteLLMModel

# Подключение к локальной модели через Ollama (используется API, совместимое с OpenAI)
model = LiteLLMModel(
    model_id="ollama_chat/qwen2.5-coder",
    api_base="http://localhost:11434",
    api_key="ollama" # Ключ не требуется, но параметр обязателен для LiteLLM
)
```

**Шаг 3. Создание кастомного инструмента (Tool)**
Инструменты — это обычные функции Python. Мы добавим встроенный инструмент поиска и создадим свой собственный для сохранения результатов.
```python
from smolagents import tool
import os

@tool
def save_report(content: str, filename: str) -> str:
    """Сохраняет текстовый контент в файл.
    Args:
        content: Текст для сохранения.
        filename: Имя файла (например, 'report.md').
    """
    with open(filename, 'w', encoding='utf-8') as f:
        f.write(content)
    return f"Отчет успешно сохранен в файл {os.path.abspath(filename)}"

# Собираем инструменты вместе
tools = [DuckDuckGoSearchTool(), save_report]
```

**Шаг 4. Инициализация и запуск агента**
```python
# Создаем агента, передавая ему модель и инструменты
agent = CodeAgent(
    tools=tools, 
    model=model,
    max_steps=5, # Максимальное количество шагов в цикле (для предотвращения бесконечных циклов)
    verbosity_level=2 # Подробный вывод для наблюдения за процессом рассуждения
)

print("Запуск агента-исследователя...")
# Даем агенту высокоуровневую задачу
task = """
1. Найди в интернете информацию о последних достижениях в области твердотельных аккумуляторов (solid-state batteries) за 2025-2026 годы.
2. Сделай краткую выжимку (3-4 абзаца).
3. Сохрани эту выжимку в файл 'battery_research.md'.
"""

result = agent.run(task)
print("\nФинальный результат:")
print(result)
```

**Шаг 5. Запуск и наблюдение за циклом (Agent Loop)**
При запуске скрипта (`python research_agent.py`) вы увидите, как агент работает в цикле:
1.  **Thought (Мысль):** Агент решает, что сначала нужно использовать инструмент поиска.
2.  **Action (Действие):** Вызывает `DuckDuckGoSearchTool` с нужным запросом.
3.  **Observation (Наблюдение):** Получает результаты поиска.
4.  **Thought:** Решает, что информации достаточно, формирует текст и вызывает инструмент `save_report`.
5.  **Action:** Сохраняет файл.
6.  **Observation:** Получает подтверждение сохранения и завершает работу.

Этот пример демонстрирует базовый принцип работы ИИ-агентов: предоставление автономности в выборе инструментов для достижения поставленной цели.
